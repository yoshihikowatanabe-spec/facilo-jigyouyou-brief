前提知識

事業用不動産の商慣習 — 購入クラウドとの違いを理解する

事業用不動産の広告コピーを書くには、居住用（購入クラウド）とは異なる業界特有の商慣習を理解する必要がある。ここでは、コピーの方向性に直結する商慣習と、頻出する業界用語を整理する。

なぜこのセクションが必要か：事業用と居住用（購入クラウド）では、「マイページ」「顧客ログ」といった同じ機能名でも、使い方・ユーザー・意味合いが全く異なる。この違いを理解しないと、購入クラウドのコピーを「投資家」に置き換えただけの訴求になってしまう。

購入クラウドとの機能対応表

機能名 購入クラウド 事業用クラウド コピーで意識すべき差
マイページ 個人の買主が物件を閲覧する画面。反響対応の効率化が目的 投資家・法人に1to1でクローズ提案する業務ツール。入札/パスの回答もここで行う 「物件探し」ではなく「クローズ提案」「非公開物件」「御社限り」の文脈で訴求
顧客ログ 個人の買主がどの物件を見たか（顧客軸）。温度感を把握して追客タイミングを見る この物件に何社が反応したか（物件軸）。資料請求数・CA請求数・入札状況を物件単位で可視化 「温度感」「追客」ではなく「出口の可視化」「失注防止」「物件サマリー」で訴求
物件登録 REINS・ポータルから取込。標準化された情報フォーマット 集約PFがない。PDF概要書を受領→自社書式に書き換え。各社バラバラの書式 「REINS連携」ではなく「概要書の書き換え」「転記作業からの解放」で訴求
マッチング 顧客の希望条件に合う物件を提案（条件登録→自動マッチ） 営業メモ（非構造化テキスト）からAIがニーズを読み取り、物件と照合 「希望条件マッチ」ではなく「記憶に頼らない」「営業メモから自動で」で訴求

事業用特有の商慣習（コピーのヒント）

🔒

非公開物件とクローズ提案

事業用の提案は「御社限り」が基本
事業用不動産では、まだ市場に出回っていない物件情報（非公開物件）を、特定の投資家・法人にだけ提案するのが一般的。「御社限りでお送りします」が営業の常套句。

メールで送ると転送されるリスクがあるが、マイページならクローズ性を担保しながら提案できる。これは購入クラウドの「物件を広く見せる」マイページとは真逆の発想。

取引形態には相対取引（1to1で個別に提案）と入札（複数社に同時提案して買い付けを競わせる）がある。提案の仕方が全く異なり、コピーの切り口も変わる。
📄

CA（秘密保持契約）と段階的な情報開示

居住用にはない「資料を見せる前の契約」が存在する
事業用では、物件の詳細資料（レントロール・収支表等）を開示する前に、CA（Confidentiality Agreement＝秘密保持契約）を締結する商慣習がある。居住用にはない概念。

つまり事業用の提案には「段階」がある：概要情報の提示 → CA締結 → 詳細資料の開示 → 入札/買付。マイページ上でこの段階管理ができることが、メールとの決定的な差になる。
📊

レントロールと重い資料

メール容量制限にひっかかるのが日常
事業用の取引では、物件概要書に加えてレントロール（賃料一覧表）・収支表・図面・登記簿・環境調査報告書など、大量の資料を共有する必要がある。居住用の「間取図1枚」とは情報量が桁違い。

メール添付ではファイルサイズの上限に頻繁にひっかかる。「レントロールをメール添付したら容量オーバーで返ってきた」は事業用仲介の営業が日常的に経験するペイン。マイページでの一元共有がこれを解消する。
💰

少数高額案件と失注の致命性

1件の他決で数千万円の手数料が飛ぶ
居住用の仲介は「数をこなす」ビジネス。事業用は1件あたり数億〜数十億円で、仲介手数料も1件で数百万〜数千万円になる。

つまり1件の失注が致命的。「提案漏れ」「フォローの遅れ」「誰がどこまで提案したかわからない」が、居住用とは桁違いのインパクトを持つ。顧客ログや提案管理の訴求は、この「1件の重み」を前提にすると刺さる。
📝

概要書の書き換え文化

ポータルもレインズもない世界でのPDFベースの情報交換
居住用にはSUUMO・レインズなどの集約プラットフォームがあるが、事業用にはそれがない（一部のオフィスPFを除く）。物件情報は各社が作成した物件概要書（PDF/Excel）で直接流通する。

受領した概要書を自社の書式に書き換える作業が日常的に発生し、1件あたり数十分かかる。「概要書の転記に半日」は商談で頻出するペイン。AI自動書き換え機能はこのど真ん中を解決する。

用語解説

用語 意味 コピーでの使い方
非公開物件 まだ市場に公開されていない物件情報。限られた先にだけ提案される 「非公開物件の提案、メールで送って大丈夫ですか？」
CA（秘密保持契約） Confidentiality Agreement。詳細資料の開示前に締結する契約 「CA締結から資料開示まで、マイページで完結」
レントロール 賃貸物件の賃料一覧表。各部屋の賃料・契約期間・入居者情報等をまとめたもの 「レントロール・収支表、メール添付の容量制限にひっかかっていませんか？」
相対取引 売主と買主が1to1で交渉する取引形態。非公開・クローズが前提 「相対取引のクローズ性を保ちながら、デジタルで提案」
入札 複数の買主候補に同時に提案し、買付価格を競わせる取引形態 「入札案件の反応、電話で追いかけていませんか？」
物件概要書 物件の基本情報をまとめた書類（PDF/Excel）。各社独自の書式で作成 「概要書の書き換え、まだ手作業ですか？」
物件軸ログ 「この物件に何社が反応したか」を物件単位で可視化する機能（事業用プラン限定） 「3億の物件、5社に提案して何社が見た？」
帯替え 物件概要書を自社の書式・デザインに変換すること。「書き換え」と同義 「帯替え作業、AIが数秒で完了」
新機能

クリエイティブの主軸となる2つの新機能

この2つの機能により、事業用クラウドは「事業用仲介ツールとして胸を張って言えるレベル」になる（向井さん談）。

NEW FEATURE 01

物件概要書 AI自動書き換え

受領した物件概要書（PDF/Excel/画像）をアップロードするだけで、AIが40項目以上を自動で読み取り、自社の書式に変換して登録する機能。

解決するペイン
事業用仲介では、受領した概要書を「自社フォーマットに書き換える」作業が日常的に発生する。1件あたり数十分かかる手作業で、仕入れ担当の時間を大きく圧迫している。
単なる「取込」ではない
OCRで読み取って登録するだけでなく、自社の書式・項目体系に合わせて再構成するのがポイント。レメティス（REMETIS）はこの機能単体で導入が決まるケースがあるほど、事業用仲介のど真ん中のペイン。
訴求のキーワード
「書き換え」「自社書式に変換」「転記作業からの解放」「数十分→数秒」
NEW FEATURE 02

非構造化AIマッチング

営業の商談メモ・顧客ヒアリングメモなど、整理されていないテキストデータをAIが読み取り、物件情報と顧客ニーズを自動で照合する機能。

解決するペイン
事業用仲介では「あの物件を誰に出すか」が営業個人の記憶と人脈に依存している。ベテラン営業の頭の中にしかないマッチング判断を、AIが仕組み化する。
「共有」ではなく「マッチング」
この機能が解くのは「情報共有」の問題ではなく、「物件と顧客の照合」の問題。普段の商談メモを書くだけで、特別な入力・整理なしにAIがマッチングを行う。
訴求のキーワード
「記憶に頼らない」「メモを書くだけで」「提案漏れゼロ」「属人化の解消」
提供価値

「速く・漏れなく・見える」— 3つの価値軸

事業用クラウドが提供する価値は、1件の物件のライフサイクルに沿った3つの軸で整理できる。デモ設計（向井作成）の三幕構成と対応している。

第1幕

速く、準備する。

概要書が届いてから提案準備が完了するまでの時間を大幅に短縮。「良い物件は掲載準備中に決まる」のが事業用の現実。

対応機能：概要書AI自動書き換え（新機能①）、社内即時共有、公開範囲設定
第2幕

漏れなく、届ける。

顧客ニーズを構造化し、物件との照合を自動化。ベテランの記憶に頼らず、若手でも提案漏れを防ぐ。

対応機能：非構造化AIマッチング（新機能②）、顧客リスト・タグ管理、マイページ、個別URL提案
第3幕

見える形で、回す。

提案した後の反応・状況を可視化。「あの物件、いま誰にどこまで提案してる？」に即答できる。勘と報告会ではなく、データで営業が回る状態へ。

対応機能：提案状況一覧、行動ログ・スコアリング、物件ダッシュボード、CSV出力

定量的な裏付け（デモ用）：

・添付メールの開封率 3〜4% → マイページ提案で 約40%（約10倍）

・ID/PW登録型マイページは 8割が離脱 → トークンURL方式で解消

・提案漏れ1件 × 平均成約手数料 = 事業用では数百万円の機会損失

・概要書5件の帯替え+地図作成で1時間半かかっていた作業が自動化（東洋不動産事例）

市場理解

事業用不動産の基本構造 — 居住用との違い

居住用（購入・売却・賃貸クラウド）との根本的な違いを整理する。

ワークフローの軸
居住用 =「顧客軸」（顧客 → 物件を探す）
事業用 =「物件軸」（物件 → 出す先を探す）
意思決定
居住用 = 感情・生活イメージ・家族合意
事業用 = 数字・事業計画・投資判断
情報流通
居住用 = SUUMO・レインズ等の集約PFあり
事業用 = 集約PFなし。物件概要書の直接流通
基本ドキュメント
居住用 = 販売図面（マイソク）
事業用 = 物件概要書（物元作成→仲介が自社書式に書き換え）

事業用仲介の核心課題：集約プラットフォームがないため、物件情報は各社がExcel・紙・メールで個別管理。「あの物件を誰に出すか」は営業個人の記憶と人脈に依存しており、組織的な物件マッチングの仕組みがない。これがFacilo事業用クラウドが解こうとしている問題。

詳細

セグメント別の特性と新機能の刺さり方

🏢

オフィス

賃貸が主流（一棟売買は投資に近い）。事業用で最もボリュームが大きい主戦場
顧客像
法人の総務・管理部門。移転・拡張・統合がトリガー。テナント代表は総務部長クラスが多い
取引の特徴
坪単価×面積で比較。共益費・フリーレント・原状回復など交渉変数が多い。3〜6ヶ月が標準
営業フロー
ニーズヒアリング → 複数ビルの概要書送付 → 内覧 → 条件交渉 → 申込。物件比較表を自社書式で作成が一般的
競合環境
三鬼商事・三幸エステート・CBRE等。ビル情報DBは各社独自構築で、それ自体が競争優位
新機能の刺さり方
概要書の書き換え頻度が最も高いセグメント → 自動書き換えの最直球ターゲット。「このニーズに合うビルは？」の照合 → AIマッチングも自然にハマる
🏪

店舗

賃貸（テナント出店）。飲食・小売・サービス業。意思決定が速く、感覚値も入る
顧客像
個人事業主・中小法人のオーナー。飲食店・美容室・クリニック等。FC本部がテナント探しをするケースも
取引の特徴
立地と人通りが全て。居抜き物件は特に動きが速い。1〜3ヶ月。鮮度が命
営業フロー
空き物件情報を出店希望者にスピード配信するのが勝負。内覧→即決もあり得る
競合環境
居抜き市場・テンポスマート等の専門PFがある程度機能。事業用の中では標準化が進んでいる方
新機能の刺さり方
件数が多くサイクルが短い → 出し忘れ・マッチング漏れが実害に直結。AIマッチングで提案速度を上げる訴求が有効。概要書は簡素なため書き換え訴求の優先度はやや低い
🏭

工場

賃貸・売買ともにあり。用途地域・環境規制が絡み、専門性が極めて高い
顧客像
製造業法人の経営層・工場長。移転は全社意思決定で取締役会マター
取引の特徴
スペック要求が極めて具体的：天井高・床耐荷重・電力容量・排水設備・用途地域。合致物件が少なく6〜12ヶ月+
営業フロー
要件定義 → リストアップ → 現地調査 → 法規制確認 → 契約。工場特化のベテラン仲介者が中心
情報の属人性
最も属人度が高い。「この工場はあの会社の要件に合う」は数値照合だけでは判断不能、経験則と土地勘に大きく依存
新機能の刺さり方
属人度が最も高い → AIマッチングの価値が最大のセグメント。概要書のスペック項目も40+と多く、自動書き換えの工数削減効果も大きい
📦

倉庫

賃貸が主流。EC拡大で需要増。物流施設との境界がグラデーション
顧客像
物流会社・EC事業者・小売の物流/購買部門。3PL事業者が借り手になるケースも多い
取引の特徴
IC距離・天井高・トラックバース数・荷捌きスペースが重要。立地は「配送効率」で評価
営業フロー
オペレーション要件ヒアリング → 候補提案 → 現地確認。「地図上で見る」ニーズが強い
市場トレンド
EC拡大で慢性的な供給不足。既存倉庫の転用・リノベーション案件が増加
新機能の刺さり方
スペック適合が比較的明確なためAIマッチングが機能しやすい。概要書の書き換えニーズもあり
🚛

物流施設

売買（開発・投資）+ テナントリーシング。プレイヤーが限定的で取引単価が大きい
顧客像
デベロッパー・物流大手・機関投資家・REIT。プレイヤーが限られている
取引の特徴
取引単価が数十億〜数百億円。土地取得→開発→リーシング→売却の多段階。財務モデルが意思決定の中心
営業フロー
1案件で複数の仲介フェーズが発生し、期間1年超が普通
情報の属人性
プレイヤー少数のため人脈=情報力。情報の非対称性そのものがビジネス
新機能の刺さり方
取引件数少×単価大で1件の出し忘れの機会損失が極めて大きい。概要書は高度に専門的で項目数も多く、自動書き換えの効果大
🏘

投資用一棟マンション

売買（投資目的の一棟購入）。利回り至上主義で居住用とは全く異なる
顧客像
富裕層個人・資産管理法人・不動産投資法人。「住む」ではなく「運用する」目的。利回り・キャッシュフローで判断
取引の特徴
表面利回り・稼働率・修繕履歴が判断材料。レントロール（賃料明細）が必須書類
営業フロー
投資基準ヒアリング → 物件情報配信 → 現地確認 → 融資打診 → 買付。スピード勝負で良い物件は数日で決まる
市場の特殊性
居住用マンションと物件形態は似るが動機・評価軸・顧客属性が全く異なる。投資用専門の仲介会社が棲み分け
新機能の刺さり方
「この投資家の基準に合う物件が出たら即提案」がニーズの核心。AIマッチングは「物件→投資家」方向でハマる。概要書書き換え（レントロール統合等）もニーズあり
補足

AIマッチングの「方向」はセグメントで逆転する

事業用は基本「物件軸」（物件→出す先を探す）だが、投資用一棟は「投資家軸」（投資家の基準→物件を探す）になる。AIマッチングの訴求はセグメントごとにどちら方向のマッチングかが変わる。

セグメント マッチングの方向 コピーの方向性
オフィス 双方向（テナント↔ビル） 「このニーズに合うビルは？」「このビルに興味を持ちそうなテナントは？」

プレイヤー

事業用仲介のプレイヤー種別

「事業用仲介会社」といっても、全アセットをカバーする会社からセグメント特化まで幅がある。広告のターゲティングやコピーの解像度に直結する。

業界の前提データ

・宅建業者数は全国で132,291社（国交省、2025年3月末）。居住用・事業用の合計

・うち84%が従業者5人未満、90%超が10人未満。圧倒的に小規模事業者の業界

・オフィス仲介業者は100社以上（officee.jp）

・CBRE単体で物流賃貸仲介の面積シェア約60%（CBRE公式）

・事業用不動産仲介に特化した業界団体・公式統計は存在しない。「事業用仲介会社が何社あるか」を示す公式データはなく、以下の分類は複数ソースからの推定

🏛

大手総合（事業用部門）

全アセットタイプを横断。居住用が主力で事業用は一部門
代表的な企業
三井不動産リアルティ、東急リバブル、住友不動産販売、野村不動産ソリューションズ 等
特徴
居住用仲介が売上の柱で、事業用は専門部門。全国の支店網を活かしてオフィス・投資用・店舗を幅広く扱う
主な対象セグメント
オフィス・投資用一棟・店舗が中心。工場・物流は手薄なことが多い
Faciloとの関係
居住用クラウドの既存顧客が事業用も導入するパターン。購入・売却クラウドからのクロスセル先。ただし事業用専門のツール要件に対する解像度は高くない場合も
🏢

オフィス仲介専業

オフィスリーシングに特化。ビル情報DBが競争優位の源泉
代表的な企業
三鬼商事、三幸エステート、日本商業不動産保証 等
特徴
自社でビル情報DBを構築しており、「どのビルのどのフロアが何坪空いているか」をリアルタイムに把握している。情報量と精度が強み
概要書の扱い
扱う概要書の量が最も多い。書き換え作業は日常業務そのもの
Faciloとの関係
概要書自動書き換えの最も直接的なターゲット。独自DBを持つぶん、ツール乗り換えへの抵抗も想定される。既存の運用に「上乗せ」できるかが導入の鍵
🌐

外資系総合

大型案件・投資案件。グローバルネットワークとリサーチ力
代表的な企業
CBRE、JLL、クッシュマン＆ウェイクフィールド、コリアーズ 等
特徴
物流施設・大型オフィス・投資案件に強い。マーケットリサーチ・バリュエーション等の付加サービスをセットで提供。取引単価は大きいが件数は限定的
主な対象セグメント
物流施設、大型オフィス、投資用一棟。中小案件はスコープ外であることが多い
Faciloとの関係
グローバルのPropTech導入が進んでおり、独自のシステム基盤を持つケースが多い。Faciloのターゲットとしての優先度は現時点で低い
📈

投資用売買特化

一棟マンション・一棟ビルの投資売買に特化。スピードと情報力が勝負
代表的な企業
SRE不動産（旧ソニー不動産）、プロパティエージェント、グローバル・リンク・マネジメント 等。加えて少数精鋭の投資用専門ブティック多数
特徴
良い物件情報を最速で投資家に出すのがビジネスの核。投資家リストと物件の照合が日常業務そのもの。利回り・融資条件での絞り込みが頻繁
情報の性質
「この投資家はRC造・都内・利回り6%以上」等の条件を把握。良物件は数日で売れるため、提案の速度=成約率
Faciloとの関係
AIマッチングの最もわかりやすいターゲット。「投資家の条件に合う物件が出たら即通知」は彼らの日常業務を直接効率化する
🏭

工場・倉庫・物流特化

産業用不動産に特化。専門知識（用途地域・環境規制等）が参入障壁
代表的な企業
CBREインダストリアル、日本GLP（リーシング部門）、ロジスティクス系仲介。地場では工場専門仲介も
特徴
スペック要件の専門知識（天井高・床荷重・電力容量・環境規制）が参入障壁。プレイヤー数が少なく人脈依存が強い。取引期間が長い
概要書の扱い
概要書の項目数が多く（40項目以上）、かつ書式が各社バラバラ。書き換え工数が大きい
Faciloとの関係
属人度が最も高いセグメントを扱うため、AIマッチング・概要書書き換えの両方が深く刺さる。ただしプレイヤー数が限られ、広告でのリーチは工夫が必要
🏪

店舗・テナント仲介特化

飲食・小売のテナント賃貸に特化。居抜き市場が主戦場
代表的な企業
テンポスマート、居抜き店舗.com（M&Aプロパティーズ）、店舗そのままオークション 等のプラットフォーム系と、地場のテナント仲介会社
特徴
回転が速くスピード勝負。居抜き物件は設備付きで取引されるため、造作譲渡等の独自の商習慣がある。件数が多い
IT化の状況
テンポスマート等のPFが一定の標準化を果たしている。事業用の中では比較的IT化が進んでいるセグメント
Faciloとの関係
件数が多く提案漏れの実害が大きいためAIマッチングのニーズはある。一方、既にPFが存在するため、Faciloの差別化ポイントの明確化が必要
📍

地場・エリア密着型

特定エリア×複数アセット。オーナーとの直接関係が最大の武器
特徴
特定エリアのビルオーナー・地主との長年の関係がビジネスの基盤。オフィス・店舗・倉庫等をエリア内で横断的に扱う。5〜30人規模が多い
情報管理の実態
Excel・紙・社長の頭の中で管理。属人化の典型だが、エリアの情報密度が強み
課題
社長・ベテラン営業の退職=情報の消失。組織としての蓄積ができていない
Faciloとの関係
最もDX余地が大きいプレイヤー。属人化→組織知への転換ストーリーが直接刺さる。ITリテラシーに幅があるため「かんたんさ」の訴求も重要

注意：実際には複数タイプにまたがる企業も多い。大手総合がエリア密着的に動く支店もあれば、地場仲介が投資用に特化しているケースもある。上記は「特性が強く出る類型」として整理したもの。

商談設計

商談ターゲットの6型 — 業態×ポジションで分類

向井が作成したデモ設計から。上のプレイヤー種別（会社の類型）やセグメント（物件種別）とは別の切り口で、取引のポジションと業態で顧客を6つの型に分類する。型ごとに「3つの価値軸のどれが最も刺さるか」が変わる。

デフォルトの設計思想：事業用は物件軸の商売。デモの主語は「顧客」ではなく「1件の物件」。1枚の概要書が届いてから、提案・反応把握・報告までを1本のストーリーで見せる。
提案のキャッチコピー：「提案に、速度と精度と透明性を。」

型 説明 価値軸の重み 訴求の語彙
①収益仲介
エンド型		 投資用一棟を個人投資家・事業法人に仲介。エンド（最終顧客）比率が高い 漏れなくが最厚。マイページ・行動ログが主役。B案（マイページ軸）も有効 利回り・検討体験・買う気の見極め・「投資家の基準に合う物件が出たら即提案」
②元付
ストック型		 自社で物件を預かり（元付）、自社の顧客リストで成約を狙う 漏れなく＋見える。リストで決めきる力＋売主への報告 限定打診・鮮度維持・「レインズに出す前に自社顧客で決めきる」・売主報告
③プロ間
流通型		 業者間で物件情報を回す。帯替え（概要書の自社化）が日常業務 速くが最厚。帯替えスピード＋秘匿共有が主役 帯替え・回覧・公開範囲。「一斉配信」はNG（出回り物件の連想）
④出口直販型
（買取再販・開発）		 自社で買い取って再販、または開発後に販売。出口で仲介が発生 漏れなく（直顧客提案）＋見える（販売進捗） 出口・在庫回転・保有コスト・販売初速。仕入れ文脈は出さない
⑤銀行
連携型		 銀行からの紹介案件が売上の柱。紹介元への報告・還流が重要 見えるが最厚。CSV還流報告が主役 情報ハブ・進捗還流・紹介元への数字報告（銀行PJ語彙と統一）
⑥大手
部門型		 大手不動産会社の事業用部門。既存システムとの並存が前提 標準バランス＋導入実績文脈 既存システム併用・現場負担ゼロ・導入実績

広告クリエイティブへの示唆：この6型のうち、どの型をメインターゲットにするかで訴求コピーの方向が決まる。「速く」が刺さる型と「漏れなく」が刺さる型では、キャッチコピーの切り口が全く異なる。向井に優先ターゲットの型を確認中。

A案：物件ライフサイクル型（デフォルト）

1枚の概要書が届いてから、提案・反応把握・報告までを順に見せる。事業用仲介の標準的な業務フローに沿った構成。ほとんどの商談はこちら。

B案：マイページ軸（切替条件付き）

エンド比率が高い／顧客体験重視の場合に切替。受信体験の対比（zip添付 vs 専用URL）から入り、「なぜこの体験が作れるか」で管理画面に遡る構成。①収益仲介エンド型で有効。
